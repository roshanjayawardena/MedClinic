# Part 9: How to Test a Multi-Tenant SaaS — Architecture Tests, Integration Tests, Testcontainers

*Building a MediClinic SaaS — Part 9 of an ongoing series*

---

Eight parts in, we have a working system: patients, appointments, encounters, prescriptions, identity, billing, and notifications. Each module enforces tenant isolation, PHI rules, and role boundaries — but none of that is mechanically verified. A developer can inadvertently add a runtime dependency between modules, forget to make a handler `sealed`, or introduce a query that doesn't filter by tenant. Nothing will catch it at build time.

This article adds two testing layers:

1. **Architecture tests** — NetArchTest rules that enforce module boundaries and handler conventions at build time, before the code ever runs
2. **Integration tests** — Testcontainers spinning up a real PostgreSQL instance to verify persistence, tenant isolation, and the audit trail

Both layers are fast, deterministic, and require no external setup beyond Docker for the integration tests.

---

## Two layers, two purposes

| Layer | Tool | What it catches | Runs when |
|---|---|---|---|
| Architecture | NetArchTest | Broken module boundaries, unsealed handlers | Every build (no I/O) |
| Integration | Testcontainers + xUnit | Wrong query filters, audit trail gaps, DB schema drift | `dotnet test` (Docker required) |

Architecture tests run in milliseconds because they use reflection — no database, no network. Integration tests are slower but they exercise the actual SQL generated by EF Core against a real PostgreSQL instance, which means they catch things that mocked DbContexts never would.

## Architecture tests: enforcing the rules mechanically

### The module boundary rule

Golden rule #10: "Modules reference only each other's `.Contracts` — never the runtime project."

This rule prevents the modular monolith from drifting into a big ball of mud where any module can call into any other module's internal logic. It's easy to state and easy to forget. Architecture tests make it impossible to ignore.

```csharp
public sealed class ModuleBoundaryTests
{
    private static readonly Assembly[] ModuleRuntimeAssemblies =
    [
        typeof(Patients.PatientsModule).Assembly,
        typeof(Appointments.AppointmentsModule).Assembly,
        typeof(Encounters.EncountersModule).Assembly,
        typeof(Prescriptions.PrescriptionsModule).Assembly,
        typeof(Identity.IdentityModule).Assembly,
        typeof(Billing.BillingModule).Assembly,
        typeof(Notifications.NotificationsModule).Assembly,
    ];

    private static readonly HashSet<string> RuntimeAssemblyNames =
        ModuleRuntimeAssemblies.Select(a => a.GetName().Name!).ToHashSet();

    [Fact]
    public void Modules_ShouldNot_Reference_Each_Others_Runtime_Assembly()
    {
        var violations = new List<string>();

        foreach (var assembly in ModuleRuntimeAssemblies)
        {
            var assemblyName = assembly.GetName().Name!;

            var illegalReferences = assembly
                .GetReferencedAssemblies()
                .Select(r => r.Name!)
                .Where(name => RuntimeAssemblyNames.Contains(name) && name != assemblyName)
                .ToList();

            foreach (var illegal in illegalReferences)
                violations.Add($"{assemblyName} → {illegal}");
        }

        violations.Should().BeEmpty(
            because: "modules must communicate only through *.Contracts assemblies, " +
                     "never through each other's runtime projects");
    }
}
```

The test loads every module's runtime assembly, then checks what each assembly references. If `Billing` ever adds a project reference to `Patients` (not `Patients.Contracts`), the test fails immediately with a message like `Billing → Patients`. The developer sees this before the PR is raised.

Notice the strategy for identifying runtime assemblies: each module has a `*Module` class (e.g., `PatientsModule`) in its runtime assembly. Using `typeof(PatientsModule).Assembly` pins the test to the actual assembly without hardcoding a string path.

### The sealed handler convention

Golden rule #6: "sealed handlers, file-scoped namespaces, primary constructors throughout."

`sealed` on a handler is a signal, not just a style choice. It tells the reader: this class is a leaf — there is no subclass that overrides its behavior. Combined with source-generated Mediator (which registers handlers by concrete type), `sealed` also makes accidental polymorphism impossible.

```csharp
[Theory]
[MemberData(nameof(ModuleAssemblies))]
public void Handlers_MustBe_Sealed(Assembly assembly)
{
    var handlerTypes = assembly.GetTypes()
        .Where(t => !t.IsAbstract && !t.IsInterface)
        .Where(t => t.GetInterfaces().Any(i =>
            i.IsGenericType &&
            (i.GetGenericTypeDefinition().Name.StartsWith("IRequestHandler") ||
             i.GetGenericTypeDefinition().Name.StartsWith("INotificationHandler"))))
        .ToList();

    handlerTypes.Should().AllSatisfy(
        t => t.IsSealed.Should().BeTrue(
            because: $"{t.FullName} must be sealed (AGENTS.md golden rule #6)"));
}
```

This test runs once per module assembly (using `[Theory]` + `[MemberData]`). If a handler is added without `sealed`, the test fails and names the specific class. Adding a new module is a one-line change to `ModuleAssemblies`.

### The vertical-slice convention

Handlers must live in a `Features` namespace — not at the module root, not in a `Services` folder:

```csharp
[Theory]
[MemberData(nameof(ModuleAssemblies))]
public void Handlers_MustLive_In_Features_Namespace(Assembly assembly)
{
    var violators = assembly.GetTypes()
        .Where(t => /* is a handler */)
        .Where(t => t.Namespace is not null && !t.Namespace.Contains(".Features."))
        .Select(t => t.FullName!)
        .ToList();

    violators.Should().BeEmpty(
        because: "handlers must live in a Features subfolder per the vertical-slice convention");
}
```

Running `dotnet test tests/Architecture` takes about two seconds and covers all seven modules. Zero setup, zero I/O.

## Integration tests: a real database, not a mock

The `testing.md` conventions are explicit: **never mock `DbContext`**. The reasons are practical:

- Mocked EF Core doesn't execute LINQ to SQL — you're testing your in-memory LINQ, not the query the database will run
- Query filters (`HasQueryFilter`) don't apply on in-memory mocks — this is the exact mechanism that enforces tenant isolation
- Migration drift isn't caught — a column renamed in the model but not the migration will pass mocked tests and fail in production

Testcontainers solves this by spinning up a real PostgreSQL Docker container, applying the schema, and tearing it down at the end of the test run.

### The test infrastructure

**`PostgresFixture.cs`** starts the container and creates the schema:

```csharp
public sealed class PostgresFixture : IAsyncLifetime
{
    private readonly PostgreSqlContainer _container = new PostgreSqlBuilder()
        .WithImage("postgres:16-alpine")
        .WithDatabase("mediclinic_test")
        .WithUsername("test")
        .WithPassword("test")
        .Build();

    public string ConnectionString => _container.GetConnectionString();

    public async Task InitializeAsync()
    {
        await _container.StartAsync();

        // EnsureCreated() creates the schema from the current EF model in one step.
        // MigrateAsync() replays migration history — correct for production, unnecessary here.
        await using var db = BuildPatientsDbContext(Guid.Empty);
        await db.Database.EnsureCreatedAsync();
    }

    public async Task DisposeAsync() => await _container.DisposeAsync();

    public PatientsDbContext BuildPatientsDbContext(Guid tenantId)
    {
        var options = new DbContextOptionsBuilder<PatientsDbContext>()
            .UseNpgsql(ConnectionString)
            .Options;

        return new PatientsDbContext(options, new TestTenantContext(tenantId), TimeProvider.System);
    }

    public TestDbContextFactory<PatientsDbContext> BuildPatientsDbContextFactory(Guid tenantId)
        => new(() => BuildPatientsDbContext(tenantId));
}
```

`EnsureCreated()` vs `MigrateAsync()` is a deliberate choice. In tests, you want the schema to match the current EF model — no more, no less. `MigrateAsync()` can succeed with an outdated model if migrations haven't been generated yet, masking model/migration drift. `EnsureCreated()` creates what EF currently knows about, which is what the tests are actually exercising.

**`TestTenantContext.cs`** is a simple `ITenantContext` stub:

```csharp
public sealed class TestTenantContext(Guid tenantId) : ITenantContext
{
    public Guid TenantId { get; } = tenantId;

    public static TestTenantContext New() => new(Guid.NewGuid());
}
```

Each test creates this with a fresh GUID. Because `BaseDbContext` stamps `TenantId` on every entity written, and filters every query by `TenantId`, data from one test is completely invisible to another — without any teardown or cleanup code.

**`TestDbContextFactory.cs`** wraps a factory function to satisfy the `IDbContextFactory<T>` contract that production handlers use:

```csharp
public sealed class TestDbContextFactory<TContext>(Func<TContext> factory)
    : IDbContextFactory<TContext>
    where TContext : DbContext
{
    public TContext CreateDbContext() => factory();

    public Task<TContext> CreateDbContextAsync(CancellationToken cancellationToken = default)
        => Task.FromResult(factory());
}
```

This lets tests create a handler with `new RegisterPatientHandler(dbFactory, tenantContext, TimeProvider.System)` without needing a full DI container.

### Testing persistence

```csharp
public sealed class PatientRegistrationTests(PostgresFixture fixture)
    : IClassFixture<PostgresFixture>
{
    [Fact]
    public async Task RegisterPatient_WithConsent_PersistsPatientToDatabase()
    {
        var tenantId = Guid.NewGuid();
        var dbFactory = fixture.BuildPatientsDbContextFactory(tenantId);
        var tenantContext = new TestTenantContext(tenantId);
        var handler = new RegisterPatientHandler(dbFactory, tenantContext, TimeProvider.System);

        var command = new RegisterPatientCommand(
            FirstName: "Alice", LastName: "Nguyen",
            DateOfBirth: new DateOnly(1985, 3, 12),
            ContactPhone: "+64 21 555 0101",
            ConsentToDataProcessing: true,
            ConsentToCommunications: true);

        var result = await handler.Handle(command, CancellationToken.None);

        result.IsSuccess.Should().BeTrue();

        // Verify the patient is actually in the database.
        await using var verify = fixture.BuildPatientsDbContext(tenantId);
        var saved = await verify.Patients.SingleOrDefaultAsync(p => p.Id == result.Value.PatientId);
        saved.Should().NotBeNull();
        saved!.TenantId.Should().Be(tenantId);
    }
}
```

The test constructs the handler directly with test doubles. There's no mocking framework, no DI container setup, and no test database cleanup — the new `tenantId` GUID ensures complete isolation.

### Why validators get their own tests

In this architecture, validators are `IEndpointFilter` classes — they run at the HTTP layer before Mediator is called. The handler trusts that the validator has already executed. Testing the handler for validation failures would require mocking the endpoint pipeline, which is fragile.

Instead:

```csharp
// Pure unit test — no database, no Docker
public sealed class RegisterPatientValidatorTests
{
    private readonly RegisterPatientValidator _validator = new(TimeProvider.System);

    [Fact]
    public void Validate_WithConsentFalse_Fails()
    {
        var command = new RegisterPatientCommand(
            FirstName: "Bob", LastName: "Smith",
            DateOfBirth: new DateOnly(1970, 6, 1),
            ContactPhone: "+64 21 555 0002",
            ConsentToDataProcessing: false);

        var result = _validator.Validate(command);

        result.IsValid.Should().BeFalse();
        result.Errors.Should().ContainSingle(e =>
            e.PropertyName == nameof(RegisterPatientCommand.ConsentToDataProcessing));
    }
}
```

Validators are pure functions that take a command and return a `ValidationResult`. They're the easiest things in the system to test — no database, no DI, no fixtures.

### The tenant isolation test

This is the highest-value test in the suite. A bug here means clinic B can read clinic A's patient records:

```csharp
[Fact]
public async Task Patient_RegisteredUnderClinicA_IsNotVisibleToClinicB()
{
    var clinicA = Guid.NewGuid();
    var clinicB = Guid.NewGuid();

    // Register a patient under clinic A.
    var handlerA = new RegisterPatientHandler(
        fixture.BuildPatientsDbContextFactory(clinicA),
        new TestTenantContext(clinicA),
        TimeProvider.System);

    var result = await handlerA.Handle(
        new RegisterPatientCommand(
            FirstName: "Protected", LastName: "Patient",
            DateOfBirth: new DateOnly(1990, 1, 1),
            ContactPhone: "+64 21 555 0303",
            ConsentToDataProcessing: true),
        CancellationToken.None);

    result.IsSuccess.Should().BeTrue();
    var patientId = result.Value.PatientId;

    // Clinic A can see its own patient.
    await using var dbA = fixture.BuildPatientsDbContext(clinicA);
    var visibleToA = await dbA.Patients.SingleOrDefaultAsync(p => p.Id == patientId);
    visibleToA.Should().NotBeNull();

    // Clinic B must NOT see clinic A's patient.
    await using var dbB = fixture.BuildPatientsDbContext(clinicB);
    var visibleToB = await dbB.Patients.SingleOrDefaultAsync(p => p.Id == patientId);
    visibleToB.Should().BeNull(
        because: "the global query filter must prevent cross-tenant reads");
}
```

The test creates two DbContexts with different `TenantId` values and queries the same database. The EF global query filter (`HasQueryFilter(e => !e.IsDeleted && e.TenantId == tenantContext.TenantId)`) in `BaseDbContext` does the work. The test verifies that filter is present and correctly wired — not just that it's defined in code.

This test would **fail** if:
- `base.OnModelCreating(modelBuilder)` was called BEFORE module-level configuration (overriding the filter)
- `PatientsDbContext` forgot to call `base.OnModelCreating(modelBuilder)` at all
- The `ITenantContext` was accidentally registered as a singleton and its `TenantId` didn't change per request

These are exactly the kinds of bugs that would pass code review but cause a production incident.

### The audit trail test

Golden rule #9: "Every read or write of an Encounter or Prescription MUST emit an audit entry." The Patients module applies the same pattern to patient registration:

```csharp
[Fact]
public async Task RegisterPatient_WritesAuditEntry_InSameTransaction()
{
    var tenantId = Guid.NewGuid();
    var handler = new RegisterPatientHandler(
        fixture.BuildPatientsDbContextFactory(tenantId),
        new TestTenantContext(tenantId),
        TimeProvider.System);

    var result = await handler.Handle(
        new RegisterPatientCommand(
            FirstName: "Audit", LastName: "Test",
            DateOfBirth: new DateOnly(1980, 7, 4),
            ContactPhone: "+64 21 555 0404",
            ConsentToDataProcessing: true),
        CancellationToken.None);

    result.IsSuccess.Should().BeTrue();

    await using var verify = fixture.BuildPatientsDbContext(tenantId);

    var auditEntry = await verify.AuditEntries
        .SingleOrDefaultAsync(a =>
            a.EntityId == result.Value.PatientId.ToString() &&
            a.Action == "PatientRegistered");

    auditEntry.Should().NotBeNull(
        because: "handler must write an AuditEntry in the same SaveChangesAsync call");
    auditEntry!.EntityType.Should().Be("Patient");
    auditEntry.TenantId.Should().Be(tenantId);
}
```

This test verifies the handler's actual audit write — not just that `db.AuditEntries.Add()` was called (which a mock would check), but that the row exists in PostgreSQL after `SaveChangesAsync` committed. Because both the patient and the audit entry are written in the same `SaveChangesAsync` call, they're atomic: if one fails, neither is persisted.

## Running the tests

```bash
# Architecture tests — no Docker required, runs in ~2 seconds
dotnet test tests/Architecture

# Integration tests — requires Docker; starts a container per test run
dotnet test tests/Integration
```

To run both:

```bash
dotnet test
```

The integration test container starts once per `PostgresFixture` instance (shared via `IClassFixture<PostgresFixture>`). All four Patients test classes share one container. Startup takes about three seconds on a warm Docker installation.

## What we're NOT testing here (and why)

**Event handlers** (e.g., `OnEncounterClosedHandler` in Billing) are integration-tested by publishing a real `EncounterClosedIntegrationEvent` and asserting the resulting `Invoice` row. This requires wiring up Mediator in the test fixture, which is straightforward with `ServiceCollection` but beyond the scope of Part 9.

**Endpoint filters** (the HTTP layer) are tested via `WebApplicationFactory<T>`, which starts a real test host. Again, straightforward but beyond scope.

**Multi-module flows** (book appointment → close encounter → create invoice → notify patient) are best tested as end-to-end tests with a running API. That's Part 10 territory.

## What makes this test suite reliable

| Property | How it's achieved |
|---|---|
| No shared state between tests | Fresh `tenantId` per test — no cleanup needed |
| Real SQL, not LINQ simulation | Testcontainers with PostgreSQL 16 |
| Query filters proven to work | Tenant isolation test fails if `base.OnModelCreating` is missing |
| Audit trail proven to be atomic | AuditTrail test fails if entries are written outside `SaveChangesAsync` |
| Boundary rules mechanically enforced | Architecture test fails on first illegal assembly reference |
| Handler conventions enforced | Architecture test fails on first unsealed or misplaced handler |
